##1. mavenu导包出现报错Perhaps you are running on a JRE rather than a JDK?

  • 出现原因：maven打包需要jdk环境而不是jre环境

  • 解决方案

    • 方案一：在maven的setting.xml 文件 标签中加入指定JDK的版本

    • 方案二：eclipse修改Installed JRES 位置在–>【Window】–>【Prefrences】–>【Java】- 【Installed JREs】

    • 方案三：到你的项目，右击Build Path->Configure Build Path；
      看Libraries里边是否引用了JavaSE-1.8，如果没有，则点右侧Add Library，按下面图片进行选择（需要把之前的JRE System Library删掉）；

      ###注意点：如果以上方法没有解决，请在方案二的时候进入子目录选择对应版本jdk，因为部分eclipse版本不支持自动修改。

##2. xml解析
  常见的有四种解析方式：DOM ,SAX,DOM4J,JDOM

    DOM方式：DOM是以层次结构组织的节点或信息片断的集合。这个层次结构允许开发人员在树中寻找特定信息。分析该结构通常需要加载整个文档和构造层次结构，然后才能做任何工作。由于它是基于信息层次的，因而DOM被认为是基于树或基于对象的。DOM以及广义的基于树的处理具有几个优点。首先，由于树在内存中是持久的，因此可以修改它以便应用程序能对数据和结构作出更改。它还可以在任何时候在树中上下导航，而不是像SAX那样是一次性的处理。

    SAX方式：SAX解析器采用了基于事件的模型，它在解析XML文档的时候可以触发一系列的事件，当发现给定的tag的时候，它可以激活一个回调方法，告诉该方法制定的标签已经找到。SAX对内存的要求通常会比较低，因为它让开发人员自己来决定所要处理的tag.特别是当开发人员只需要处理文档中所包含的部分数据时，SAX这种扩展能力得到了更好的体现。但用SAX解析器的时候编码工作会比较困难，而且很难同时访问同一个文档中的多处不同数据。一般来说，SAX还比它的替代者DOM快许多。

    JDOM方式 JDOM的目的是成为Java特定文档模型，它简化与XML的交互并且比使用DOM实现更快。JDOM与DOM主要有两方面不同。首先，JDOM仅使用具体类而不使用接口。这在某些方面简化了API，但是也限制了灵活性。第二，API大量使用了Collections类，简化了那些已经熟悉这些类的Java开发者的使用。

    DOM4J方式 DOM4J是一个非常非常优秀的Java XML API，具有性能优异、功能强大和极端易用使用的特点，同时它也是一个开放源代码的软件。如今你可以看到越来越多的Java软件都在使用DOM4J来读写XML，特别值得一提的是连Sun的JAXM也在用DOM4J。

###总结：

  • DON
    • 优点： 形成树结构，易编写，易理解，树结构保存在内存中易修改，效率高
    • 缺点： 文件较大是对内存损耗大，容易出现溢出，将整个文件调入内存，浪费空间
  • SAX
    • 优点： 事件驱动模式，对内存耗费小 可以只处理XML中的数据
    • 缺点： 编码难度大，不能实现访问同一XML多处不同的数据，访问效率低
  • JDOM
    • 优点： 易编码 易上手 没有向下兼容的限制，因此比DOM简单
    • 缺点： 性能差 不能处理大于内存的文档
  • DOM4J
    • 优点： 性能优异 灵活性好 功能强大 易编码
    • 缺点 API比较复杂

##5. 开发注意点和小知识点
+ 注意文件编码格式&ensp;&ensp;&ensp;最好别用记事本打开一些文件，编码错误问题不容易被发现。
+ 注意代码编码格式&ensp;&ensp;&ensp;去除一些无用，不清楚原因的空格等小细节问题。
+ 注意启动服务&ensp;&ensp;&ensp;&ensp;&ensp;&ensp;&ensp;关闭之前启动的窗口        

+ Linux端口常见操作命令 
 + netstat -ntlp //查看当前所有tcp端口·
 + netstat -nulp //查看所有UDP端口
 + netstat -nlp  //查看端口占用
 + netstat -p    //查看端口进程（应用服务）
 + netstat -ntulp |grep 80  //查看所有80端口使用情况·
 + netstat -anp | grep 3306  //查看所有3306端口使用情况·
 + ps -ef | grep xxx(某个进程) //可以查看某个进程的pid。 
 + kill -9 $pid //释放端口
 + sudo fuser -k -n tcp 80 //强制杀死该端口进程
